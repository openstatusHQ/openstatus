Openstatus is built for engineers. Everything the dashboard does — managing uptime monitors, publishing status pages, posting status reports, scheduling maintenance — is also reachable from your terminal, CI pipeline, Terraform plan, and AI assistant. Same workspace, same API key, same audit log.

## CLI

The openstatus CLI lets you manage monitors, status pages, and incidents directly from your terminal. It's designed to work for both humans (interactive wizards, colors, spinners) and agents (`--json`, deterministic flags, structured errors).

```bash
brew install openstatusHQ/cli/openstatus --cask
openstatus whoami
openstatus monitors apply
openstatus status-report create
```

Every command supports `--json` for scriptable output and `--quiet` for CI pipelines. Required flags fail with a clear list of what's missing instead of hanging on a TTY check.

Read more in the [CLI reference](https://docs.openstatus.dev/reference/cli-reference/).

## API

The openstatus API is a typed, JSON-over-HTTP layer powered by [ConnectRPC](/blog/migrating-from-zod-openapi-to-connectrpc) — every endpoint the dashboard uses, with generated clients on top. Authenticate with the `x-openstatus-key` header, the same key the CLI, SDK, Terraform provider, and MCP server share.

```bash
curl https://api.openstatus.dev/rpc/openstatus.v1.MonitorService/ListMonitors \
  -H "x-openstatus-key: os_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx" \
  -H "Content-Type: application/json" \
  -d '{}'
```

Generate a key from **Settings → API Tokens**. Every mutation through the API is workspace-scoped and audit-logged.

Read more in the [API reference](https://api.openstatus.dev/openapi).

## Node SDK

The [openstatus Node SDK](https://github.com/openstatusHQ/sdk-node) is a fully typed TypeScript and JavaScript client for the API — no hand-written `fetch` calls, no codegen step. Works in Node and Deno.

```bash
npm i @openstatus/sdk-node
```

And use it like:

```ts
import { Openstatus } from "@openstatus/sdk-node";

const client = new Openstatus({ apiKey: process.env.OPENSTATUS_API_KEY });

const monitors = await client.monitors.list();
await client.statusReports.create({ /* ... */ });
```

Read more in the [SDK reference](https://jsr.io/@openstatus/sdk-node) — monitors, status pages, status reports, maintenances, notifications, and health. [Reach out](mailto:ping@openstatus.dev) if you want us to port it to another language.

## Terraform

The [openstatus Terraform provider](https://registry.terraform.io/providers/openstatusHQ/openstatus/latest) treats monitoring as code: monitors, notifications, and status pages live in the same plan-and-apply lifecycle as the rest of your infrastructure.

```hcl
terraform {
  required_providers {
    openstatus = {
      source = "openstatusHQ/openstatus"
    }
  }
}

resource "openstatus_monitor" "api" {
  name      = "API"
  url       = "https://api.example.com/health"
  frequency = "1m"
  regions   = ["iad", "ams", "syd"]
}
```

Bootstrap an existing workspace with `openstatus terraform generate`, which fetches your full configuration and produces valid HCL with cross-references and import blocks — no hand-writing required.

See the [Terraform reference](https://docs.openstatus.dev/reference/terraform/) for the full list of resources.

## MCP Server

The openstatus MCP server connects Claude Desktop, ChatGPT, Cursor, and any other [Model Context Protocol](https://modelcontextprotocol.io) client to your workspace — so an AI assistant can read and update status pages, status reports, and maintenance windows directly from a conversation.

```json
{
  "mcpServers": {
    "openstatus": {
      "url": "https://api.openstatus.dev/mcp",
      "headers": {
        "x-openstatus-key": "os_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx"
      }
    }
  }
}
```

The server is stateless Streamable HTTP and exposes 8 tools scoped to your workspace. Every mutation requires the assistant to explicitly choose whether to notify subscribers — so an LLM cannot quietly fan out an alert. Every call lands in the audit log under `actor_type = 'mcp'`.

Authentication is the `x-openstatus-key` header — the same API key the CLI, API, and Terraform use. We don't implement the MCP OAuth flow: there's no separate MCP credential to manage, no consent screen, just one key from **Settings → API Tokens**.

Read the [MCP reference](https://docs.openstatus.dev/reference/mcp-server/) for the full tool list and configuration.

## Skills

[Openstatus skills](https://github.com/openstatusHQ/skills) are agent-ready bundles that wrap the CLI for terminal-shaped AI workflows like Claude Code. Install once, then let your agent draft incidents, schedule maintenance, and post status updates on your behalf.

```bash
npx skills add openstatushq/skills
```

What you get: incident communication templates, maintenance announcement drafts, periodic status reports, and shared status-page context so every skill speaks in your voice.

## Slack Agent

If your team lives in Slack, `@openstatus` in any channel or thread to create, update, and resolve incidents in plain language — no slash commands, no context switching. The agent shows a confirmation card before anything goes public, and every approved mutation lands in the audit log under `actor_type = 'slack'`. Read more on the [status page product page](/status-page#slack-agent).
